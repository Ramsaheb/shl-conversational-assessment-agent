Approach Document: SHL Conversational Assessment Recommender

Ramsaheb Prasad  |  AI Intern Take-Home Assessment  |  May 2026  |  Stack: FastAPI, ChromaDB, Groq (Llama 3.1 8B), Sentence-Transformers

1. Architecture & Design Choices

The system is a stateless FastAPI microservice with a deterministic orchestration layer — no LangChain, no LangGraph. The LLM generates only natural language; all routing, safety, and grounding decisions are made in Python.

User Message → POST /chat ↓ Conversation State Extractor (deterministic parsing) ↓ Intent Detector (refusal → comparison → refinement → recommendation → clarification) ↓ Hybrid Retriever (ChromaDB semantic + Jaccard keyword) ↓ Ranker (semantic 60% + keyword 30% + metadata boost 10% + domain penalty) ↓ LLM Summary (Groq: llama-3.1-8b-instant) ↓ Post-Processor (URL sanitization + name grounding + catalog validation) ↓ Pydantic Response {reply, recommendations[], end_of_conversation}
  • Stateless: Full conversation history is sent with every request. State (role, seniority, skills, assessment type needs) is extracted per-request via deterministic keyword/regex parsing.
  • Deterministic Intent Routing: Six intents (refusal, comparison, refinement, recommendation, greeting, clarification) detected via Python rules, not LLM classification. This prevents non-deterministic routing loops and ensures predictable behavior.
  • Turn-Cap Enforcement: A hard limit forces best-effort recommendation when the conversation approaches 8 turns, preventing infinite clarification loops.
  • Safety-First Refusals: A dedicated refusal service runs before any LLM call, using regex patterns for prompt injection and keyword matching for off-topic detection (legal, salary, competitors, general hiring advice).

2. Retrieval Strategy

  • Catalog: 377 Individual Test Solutions scraped from the SHL product catalog via a custom BeautifulSoup pagination scraper. Each item stores name, URL, test type code (A/B/C/K/P/S), description, skills, and tags.
  • Embedding Model: sentence-transformers/all-MiniLM-L6-v2 (runs locally, fast, good semantic quality for short text).
  • Hybrid Search: Broad semantic retrieval (top 30 from ChromaDB) combined with Jaccard keyword overlap scoring — no hard metadata filtering.
  • Smart Ranking: Combined score = 60% semantic + 30% keyword + 10% metadata type bonus. A domain-mismatch penalty demotes results whose names contain unrelated tech keywords (e.g., "Java Web Services" for a customer service query).
  • No Hard Filtering: Early experiments with metadata-only filtering (e.g., only type "K" for developers) destroyed Recall@10 by excluding valid cognitive/personality assessments. Soft boosting preserves recall while improving precision.

3. Prompt Design & Grounding

  • Post-Retrieval Generation: The LLM receives only validated catalog items and is asked to explain why they fit. It never generates names, URLs, or types from its own weights.
  • URL Sanitization: A regex post-processor strips any URLs the LLM may inject into the reply text, ensuring URLs appear only in the structured recommendations array.
  • Name Grounding: A whitelist checker replaces any assessment name in the reply that isn't in the current shortlist with generic text ("a relevant assessment"), preventing hallucinated references.
  • Catalog Validation: Every recommendation is cross-referenced against the full catalog before returning. Non-matching items are silently dropped.

What Didn't Work:

  • Initial catalog had only ~22 hand-curated items → destroyed Recall@10. Fixed by scraping the full 377-item catalog.
  • Hard metadata filtering excluded valid cross-domain assessments → replaced with soft ranking boost.
  • Legal queries like "Is it legal to use personality tests?" bypassed refusal because only "legal advice" (exact phrase) was matched → expanded to include "is it legal", "compliance", "EEOC", "regulations".
  • Comparison for "OPQ vs Verify G+" failed because alias matching didn't map abbreviations → added a dedicated alias dictionary for common short-forms.
  • Keyword "service" in "customer service" matched "Java Web Services" → added domain-mismatch penalty in ranking.

How I Measured Improvement: Each fix was validated by re-running the full behavioral probe suite (8 core scenarios + 9 edge cases) and checking Recall@10 against the 10 public traces. The probe pass-rate went from 11/17 to 17/17 across iterations. Latency was tracked per-call to stay under the 30-second evaluator timeout.
